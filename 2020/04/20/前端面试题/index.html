移动端适配相关

iphone6分辨率是750*1334，750是他的物理像素，375是逻辑像素，像素比是 物理像素/逻辑像素 所以等于2。

css中的width对应的其实是逻辑像素，比如说写个 width:375px; 就可以铺满iph6。

但是每个手机像素不一样，如果固定设置375px，它在300像素的手机上就会出现滚动条，所以需要通过rem动态设置。

目前移动端的设计图都是按照375逻辑像素的ip6来设计的，但是为了展示更细节的东西，用的是750的psd图。

REM就html元素的font-size。HTML元素的font-size:16px; 1rem=16px;

方案一：JS配合修改配合rem适配

屏幕宽度与设计稿宽度相同时(scale为1)设置根元素字体大小为100px，那么接下来只需要考虑其他屏幕宽度/设计稿宽度的比例。

方案二：

将rem设置为当前屏幕宽度的16分之一。

你也可以改成n分之一，无所谓的。

怎么实现1rem表示为50px？

当设计稿宽度与屏幕宽度相同时rem为50px

其他宽度：

​ rem = 屏幕宽度/设计稿宽度 * 50

闭包

因为JS作用域链的存在，函数作用域内部可以访问函数上级作用域的变量/数据，但是函数外部不能访问函数内部的变量，这时候就需要闭包。可以说闭包就是能够访问其他函数内部的变量/数据。

闭包更深层：

​ JS垃圾回收机制会在变量被使用过后回收，但是闭包保存了外部函数的变量，导致变量一直存在，所以我们能够访问他，但是相对的也增大了内存的使用。又因为变量一直处于函数中，所以它很难被污染，所以闭包给我们提供了一个干净而且长久存在的空间来保存变量。

vue的计算属性、方法、侦听属性

计算属性用来处理复杂逻辑

计算属性VS方法

你会发现计算属性实现的使用方法也能实现，两者的区别是什么？

不同的是计算属性是基于它们的响应式依赖进行缓存的。只在相关响应式依赖发生改变时它们才会重新求值。这就意味着只要 message 还没有发生改变，多次访问 reversedMessage 计算属性会立即返回之前的计算结果，而不必再次执行函数。

相比之下，每当触发重新渲染时，调用方法将总会再次执行函数。

计算属性VS侦听器（watch）

watch适合处理的场景是，侦听一个数的变化，当该数据变化，来处理其他与之相关数据的变化（该数据影响别的多个数据）
computed适合处理的场景是，获得一个值或者结果，该结果受其他的依赖的影响。（一个数据受多个数据影响）

虚拟DOM是什么

virtual dom其实是以JavaScript对象作为基础的树，用对象属性来描述节点，它是对于真实DOM的抽象。它保存了被渲染之前包含的所有信息，并且可以通过js操作进行维护。

优点：

  1. 跨平台优势
  2. 操作DOM慢，JS运行效率高，可以将虚拟DOM放在JS层，提高效率
  3. 提升渲染性能

vue生命周期

vue生命周期

beforeCreate

​ 初始化了一个vue空的实例对象，这时候只有默认的生命周期和默认的事件

​ 此时数据没初始化好，无法访问数据

*created *

​ 数据已初始化完毕，可以被访问，是渲染之前修改数据的机会

beforeMount

​ 模板已经在内存中编译好，但是没有挂在到页面中去，此时完成了虚拟DOM的构建，但并未渲染，最后一次修改数据的机会

mounted

​ 模板已经挂载完毕，完成渲染，可以操作DOM

beforeUpdate

​ 在重新渲染之前调用，页面中的数据是旧的，实例数据是最新的，页面尚未和最新数据同步

updated

​ 在重新渲染之后调用，已经渲染完成，页面更新，实例与页面数据同步

beforeDestroy

​ VUE实例被摧毁之前被调用，这时候实例上的数据、过滤器、事件都可以使用，还没到真正摧毁的阶段，

可以用来清除计时器或事件

destroyed

组件已经被完全被摧毁，组件中的事件、数据、方法、指令都不能用了

VUE单页应用(SPA)开发有什么优点

优点(交互体验好，前后端分离，服务器压力小)

缺点（首次加载慢，不利于SEO搜索引擎，不适合大型项目开发）

一、优点(交互体验好，前后端分离，服务器压力小)

1.良好的交互体验

　　单页应用的内容的改变不需要重新加载整个页面，获取数据也是通过Ajax异步获取，没有页面之间的切换，就不会出现“白屏现象”,也不会出现假死并有“闪烁”现象，页面显示流畅，web应用更具响应性和更令人着迷。

2.良好的前后端工作分离模式

　　后端不再负责模板渲染、输出页面工作，后端API通用化，即同一套后端程序代码，不用修改就可以用于Web界面、手机、平板等多种客户端。

3.减轻服务器压力

　　单页应用相对服务器压力小，服务器只用出数据就可以，不用管展示逻辑和页面合成，吞吐能力会提高几倍。

二、缺点（首次加载慢，不利于SEO搜索引擎，不适合大型项目开发）

1.首屏加载慢

  • 如果不对路由进行处理，在加载首页的时候，就会将所有组件全部加载，并向服务器请求数据，这必将拖慢加载速度；
  • 通过查看Network，发现整个网站加载试讲长达10几秒，加载时间最长的就是js、css文件和媒体文件及图片

　　解决方案：

  • Vue-router懒加载

　　　　Vue-router懒加载就是按需加载组件，只有当路由被访问时才会加载对应的组件，而不是在加载首页的时候就加载，项目越大，对首屏加载的速度提升得越明显。

  • 使用CDN加速

　　　　在做项目时，我们会用到很多库，采用cdn加载可以加快加载速度。

  • 异步加载组件

　　　　这里可以参考别人的介绍（https://segmentfault.com/a/1190000012138052）

  • 服务端渲染

　　　　服务端渲染还能对seo优化起到作用，有利于搜索引擎抓取更多有用的信息（如果页面纯前端渲染，搜索引擎抓取到的就只是空页面）

2.不利于SEO

　　seo 本质是一个服务器向另一个服务器发起请求，解析请求内容。但一般来说搜索引擎是不会去执行请求到的js的。也就是说，搜索引擎的基础爬虫的原理就是抓取url，然后获取html源代码并解析。 如果一个单页应用，html在服务器端还没有渲染部分数据数据，在浏览器才渲染出数据，即搜索引擎请求到的html是模型页面而不是最终数据的渲染页面。 这样就很不利于内容被搜索引擎搜索到。

　　解决方案：

  • 服务端渲染

　　　　服务器合成完整的 html 文件再输出到浏览器

  • 页面预渲染
  • 路由采用h5 history模式

3.不适合开发大型项目

　　大型项目中可能会涉及大量的DOM操作、复杂的动画效果，也就不适合使用Vue、react框架进行开发

后台管理怎么做的权限（我只有token，就是简单的登录权限）

默认token顺序

  • 1.处理注册登录问题，用户模型密码需要加密
  • 2.登录时传给后端校验账号密码是否正确，正确就返回前端一个token，token通过jwt形成
  • 3.前端将token保存到本地

下面就是进行访问权限，分为http访问进行token校验、访问非登录页面也验证token

  • 1.http访问后端时，给请求表头添加本地的token，后端进行token校验，
    先从请求表头提取出token，然后逆向推出token的id，接着通过id查找用户是否存在
    一切都成功就返回数据，不成功都返回401状态码
    然后通过http的拦截器，给所有返回状态码为401的http请求都跳转到登录页面
  • 2.访问非登录页面的限制：
    给登录页面设置一个meta属性 meta: { isPublic: true }
  访问任意页面时判断是否存在token并且非登录页面，只要有一个不存在就跳转到登录页面
